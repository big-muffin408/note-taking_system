# 基于 AI 增强的协作式文档笔记系统开发者说明文档

本文面向接手本项目的开发者，说明当前代码结构、运行方式、服务边界、关键数据流、接口约定和常见开发注意事项。项目当前是一个可运行的 monorepo：前端使用 React + Vite + TypeScript，后端由多个 Node.js 微服务和一个 Python FastAPI AI 服务组成，基础设施通过 Docker Compose 编排。

## 1. 项目定位

本项目是一个面向文档学习和知识整理场景的协作式笔记系统，核心能力包括：

- 用户注册、登录、Google OAuth 登录和管理员能力。
- TipTap 富文本编辑器，支持自动保存、Markdown 导出、HTML 导出。
- MongoDB 持久化笔记 CRUD。
- Yjs + WebSocket 实时多人协同编辑。
- IndexedDB 离线缓存、离线变更队列和冲突处理。
- PDF 上传、对象存储、PDF 解析并自动生成可编辑笔记。
- AI 摘要、润色、基于 LlamaIndex + Chroma 的 RAG 问答。
- 版本历史、自动快照和版本恢复。
- 分享权限，包括只读和可写协作。

当前实现偏向“轻量可演示且链路完整”：AI provider 默认使用 `mock`，PDF 解析默认优先 MinerU，但在没有 MinerU API 或本地 `mineru` 命令时可回退到 PyMuPDF。

## 5. 本地运行方式

### 5.1 一键 Docker 完整启动

最推荐的完整验证方式：

```bash
cp .env.example .env
docker compose up -d --build
docker compose ps
```

访问入口：

- 前端和网关：`http://localhost`
- MinIO 控制台：`http://localhost:9001`

注意：`docker compose up --build` 是前台启动，终端退出后服务会停止。希望服务保持运行时使用 `-d`。

### 5.2 Node 本地开发启动

本地开发时可先启动数据库和基础设施：

```bash
docker compose up -d mysql mongodb redis minio
npm install
npm run dev
```

`npm run dev` 会启动：

- `@notes/web`
- `@notes/user-service`
- `@notes/document-service`
- `@notes/collab-service`
- `@notes/sync-service`

AI 服务需要单独启动：

```bash
conda env create -f services/ai-service/environment.yml
conda activate note-taking-ai
python -m pip install -r services/ai-service/requirements.txt
python -m uvicorn app.main:app --app-dir services/ai-service --host 0.0.0.0 --port 3003 --reload
```

### 5.3 MinerU API 模式

如需使用更高质量的 PDF 解析，可启动 MinerU 扩展编排：

```bash
cp .env.example .env
npm run docker:up:mineru
```

等价命令：

```bash
docker compose -f docker-compose.yml -f docker-compose.mineru.yml up --build
```

该模式会让 `ai-service` 通过 `MINERU_API_URL=http://mineru-api:8000` 调用独立 MinerU API 容器。没有 MinerU API 且容器内没有 `mineru` 命令时，`ai-service` 会回退到 PyMuPDF。

前端位于 `apps/web`，入口是 `apps/web/src/main.tsx` 和 `apps/web/src/App.tsx`。主要页面和模块：

| 路径 | 说明 |

| `pages/RegisterPage.tsx` | 注册页，包含邮箱验证码流程 |
| `pages/OAuthCallbackPage.tsx` | Google OAuth 回调页，从 hash 中读取 JWT |
| `pages/EditorPage.tsx` | 主编辑页，整合编辑器、协同、AI、PDF 上传、版本历史、分享、导出 |
| `pages/AdminPage.tsx` | 管理员页面 |
| `contexts/AuthContext.tsx` | 登录态、JWT、本地 token 恢复 |
| `contexts/NotesContext.tsx` | 笔记列表、离线缓存、同步队列和冲突处理 |
| `lib/offlineDb.ts` | IndexedDB 封装 |
| `lib/api.ts` | HTTP 和 SSE client |

| `components/VersionHistory.tsx` | 版本历史 UI |
| `components/ShareDialog.tsx` | 分享弹窗 |

### 8.1 前端 API 基地址

前端通过 `VITE_API_BASE_URL` 和 `VITE_WS_BASE_URL` 适配运行模式：

- Docker/Nginx 模式：`VITE_API_BASE_URL=""`，HTTP 请求走同源 `/api/...`。
- Docker/Nginx 模式：`VITE_WS_BASE_URL="/ws"`，WebSocket 连接到同源 `/ws/collab/:documentId`。
- 本地 Vite 模式：由 `apps/web/vite.config.ts` 的 proxy 转发 API 和 WebSocket。

### 8.2 登录态

登录成功后，JWT 会写入：

- `localStorage` 的 `notes_jwt`。
- IndexedDB 的 `auth` store。

应用启动时，`AuthContext` 会调用 `/api/user/me` 恢复当前用户。失败时清理本地 token。

### 8.3 笔记编辑和保存

`EditorPage` 的保存策略：

- 正文变更后 3 秒防抖保存。
- `Ctrl+S` 或页面按钮可立即保存。
- 在线且 note ID 不是 `local-*` 时，调用 `PUT /api/doc/notes/:id`。
- 离线或本地临时笔记时，将变更写入 IndexedDB 并加入同步队列。
- 服务端返回 409 时进入冲突状态，UI 提供“保留本地草稿”和“使用服务器版本”。

### 8.4 协同编辑

编辑页在线打开服务端笔记时会创建：

- 一个新的 `Y.Doc`。
- 一个 `WebsocketProvider`。
- 连接地址：`/ws/collab/:documentId?token=<jwt>`。

协同状态、在线人数和光标信息由 y-websocket awareness 同步。前端离线或本地临时笔记不启用协同连接。

### 8.5 AI 功能

编辑页提供：

- PDF 上传：`POST /api/doc/pdf/upload`。
- 摘要：`POST /api/ai/summary?stream=true`。
- 问答：`POST /api/ai/chat?stream=true`。
- 选中文本润色：`POST /api/ai/polish?stream=true`。

前端使用 SSE 解析 `meta`、`chunk`、`done` 三种事件。

### 9.4 Google OAuth 流程

```text
Frontend login page
  -> GET /api/user/google
  -> Google authorization page
  -> GET /api/user/google/callback?code=...&state=...
  -> user-service exchanges code for token
  -> user-service reads Google userinfo
  -> create or bind local user
  -> redirect to /auth/callback#token=<jwt>
  -> OAuthCallbackPage stores JWT and calls /api/user/me
```

没有配置 `GOOGLE_CLIENT_ID` 或 `GOOGLE_CLIENT_SECRET` 时，`GET /google` 会返回“Google 登录尚未配置”。

## 10. 文档服务

文档服务位于 `services/document-service`，入口是 `src/index.ts`。

### 10.1 职责

- 笔记列表、详情、创建、更新、删除。
- 所有者和分享权限检查。
- PDF 上传到 MinIO。
- 调用 AI 服务解析 PDF。
- 根据 PDF 解析结果创建可编辑笔记。
- 版本历史、版本快照和恢复。
- 内部文档所有权检查。

### 10.2 MongoDB 集合

MongoDB 初始化脚本在 `infra/database/mongodb/init.js`。

| 集合 | 说明 |
| --- | --- |
| `documents` | 笔记主体，包含 title/content/ownerId/sourcePdfId 等 |
| `versions` | 手动或自动版本快照 |
| `document_updates` | 协同服务写入的 Yjs update，初始化脚本未显式创建但运行时使用 |
| `pdf_assets` | PDF 原文件元数据 |
| `audit_logs` | 文档服务侧审计日志 |
| `collaboration_events` | 协同事件预留集合 |

`documents.content` 保存 HTML 内容；协同正文的实时状态以 Yjs update 形式保存在 `document_updates`。当存在协同状态时，文档服务的 `content` 更像兼容字段和非协同回退字段。

### 10.3 主要接口

| 方法 | 路径 | 说明 | 认证 |
| --- | --- | --- | --- |
| `GET` | `/health` | 健康检查 | 否 |
| `GET` | `/notes` | 当前用户自己的和分享给自己的笔记列表 | 是 |
| `GET` | `/notes/:id` | 获取单篇笔记 | 是 |
| `POST` | `/notes` | 创建笔记 | 是 |
| `PUT` | `/notes/:id` | 更新标题/内容，带 `baseUpdatedAt` 冲突检测 | 是 |
| `DELETE` | `/notes/:id` | 删除笔记，仅 owner | 是 |
| `POST` | `/pdf/upload` | 上传 PDF、解析、创建笔记 | 是 |
| `POST` | `/notes/:id/versions` | 创建版本快照 | 是 |
| `GET` | `/notes/:id/versions` | 列出版本 | 是 |
| `GET` | `/notes/:id/versions/:versionId` | 获取版本详情 | 是 |
| `POST` | `/notes/:id/versions/:versionId/restore` | 恢复版本 | 是 |
| `GET` | `/internal/check-ownership` | 内部所有权检查 | 否，内网调用 |

### 10.4 PDF 上传链路

```text
Frontend
  -> POST /api/doc/pdf/upload multipart file
  -> document-service validates auth and PDF type
  -> document-service stores raw PDF into MinIO
  -> document-service sends multipart to ai-service /pdf/parse
  -> ai-service parses PDF and indexes chunks into Chroma
  -> document-service creates documents record
  -> document-service creates pdf_assets record
  -> Frontend navigates to generated note
```

默认上传大小限制是 50 MB，Nginx 也配置了 `client_max_body_size 50m`。

## 11. AI 服务

AI 服务位于 `services/ai-service`，入口是 `app/main.py`。

### 11.1 职责

- PDF 解析。
- Markdown 转 HTML 草稿。
- 文本分块。
- 使用 LlamaIndex SentenceSplitter 分块，OpenAI 兼容 embedding 模型生成向量。
- 写入 Chroma 本地持久化存储。
- 按问题向量检索上下文。
- 调用 mock/deepseek/openai provider 生成摘要、润色和问答结果。
- 支持 SSE 流式输出。

### 11.2 PDF 解析策略

`PDF_PARSE_PROVIDER=mineru` 时：

1. 如果设置了 `MINERU_API_URL`，调用 MinerU API 的 `/file_parse`。
2. 否则如果容器内或本机存在 `MINERU_COMMAND`，执行命令行 MinerU。
3. 否则回退到 PyMuPDF。

`PDF_PARSE_PROVIDER=pymupdf` 时，强制使用 PyMuPDF。

### 11.3 Chroma 索引策略

- collection 默认名：`note_chunks`。
- embedding 模型：默认 `text-embedding-3-small`（OpenAI 兼容 API）。
- 持久化目录：`/app/chroma_data`（Docker volume）。
- 分块：LlamaIndex `SentenceSplitter`，chunk_size=512 token，overlap=50。
- metadata 包含：
  - `documentId`
  - `noteId`
  - `sourceName`
  - `text`
  - `markdown`
  - `indexedAt`

### 11.4 主要接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET` | `/health` | 返回 provider、模型、PDF parser、Chroma collection |
| `POST` | `/pdf/parse` | multipart PDF 解析、生成 markdown/html、写入 Chroma |
| `POST` | `/documents/index` | 显式索引一段文本 |
| `POST` | `/summary` | 摘要，可加 `?stream=true` |
| `POST` | `/polish` | 润色，可加 `?stream=true` |
| `POST` | `/chat` | RAG 问答，可加 `?stream=true` |

### 11.5 SSE 事件格式

流式接口返回 `text/event-stream`：

```text
event: meta
data: {"type":"summary","sources":[]}

event: chunk
data: {"chunk":"..."}

event: done
data: {"content":"完整结果"}
```

前端 `streamAI()` 会按事件类型回调 `onMeta`、`onChunk`、`onDone`。

## 12. 协同服务

协同服务位于 `services/collab-service`，入口是 `src/index.ts`。

### 12.1 职责

- 提供 `/health`。
- 处理 `/ws/collab/:documentId` WebSocket 升级。
- 校验 JWT。
- 检查用户对文档的 owner/share 访问权。
- 加载 MongoDB 中的 Yjs update。
- 使用 y-websocket 建立协同同步。
- 对 Yjs update 做防抖持久化。
- 用户断开时 flush 当前文档。
- 每 5 分钟自动保存版本快照，最后一个连接断开时也保存快照。

### 12.2 连接流程

```text
EditorPage creates Y.Doc
  -> WebsocketProvider connects /ws/collab/:documentId?token=<jwt>
  -> Nginx proxies to collab-service
  -> collab-service parses documentId
  -> collab-service verifies JWT
  -> collab-service reads documents collection and user-service share check
  -> collab-service loads document_updates
  -> setupWSConnection starts Yjs sync
  -> updates are persisted to MongoDB
```

`/ws/health` 不是有效健康检查路径。协同服务的 HTTP 健康检查是 `/health`，WebSocket 通道是 `/ws/collab/:documentId`。

## 13. 同步服务和离线机制

同步服务位于 `services/sync-service`，入口是 `src/index.ts`。

### 13.1 前端离线数据

IndexedDB 数据库名：`ai-notes-offline`，版本：`2`。

| Store | 说明 |
| --- | --- |
| `offline_notes` | 每个用户的本地笔记缓存 |
| `sync_queue` | 离线变更队列 |
| `auth` | JWT 备份 |

离线笔记状态：

- `synced`：已和服务端一致。
- `pending`：本地有待同步变更。
- `conflict`：服务端版本已变化，需要用户处理。
- `deleted`：本地删除等待同步。

### 13.2 同步锁和多标签页

前端使用：

- `localStorage` key：`notes-sync-lock`。
- `BroadcastChannel`：`sync-channel`。

这样可以避免多个标签页同时 push 队列，并在同步完成后通知其他标签页刷新本地缓存。

### 13.3 服务端接口

| 方法 | 路径 | 说明 | 认证 |
| --- | --- | --- | --- |
| `GET` | `/health` | 健康检查 | 否 |
| `GET` | `/pull?since=...` | 拉取自己的和分享给自己的笔记 | 是 |
| `POST` | `/push` | 推送 create/update/delete 变更 | 是 |

`/push` 的冲突依据是 `baseUpdatedAt`。当客户端基于的版本与服务端当前 `updatedAt` 不一致时，返回：

```json
{
  "status": "conflict",
  "serverNote": {
    "id": "...",
    "title": "...",
    "content": "...",
    "updatedAt": "..."
  }
}
```

前端会保存一份 `${noteId}__server` 的服务端副本，用于“使用服务器版本”。

## 14. 数据模型摘要

### 14.1 MySQL `users`

关键字段：

- `id`
- `email`
- `display_name`
- `password_hash`
- `role`
- `oauth_provider`
- `oauth_subject`
- `failed_login_attempts`
- `locked_until`
- `created_at`

### 14.2 MySQL `shares`

关键字段：

- `sharee_id`

典型结构：

协同状态结构：

```json
  "createdAt": "Date",
```

### 14.6 MongoDB `versions`

典型结构：


每篇文档最多保留 50 个版本。

## 15. 权限模型

### 15.1 JWT

JWT payload 至少包含：

```json
{
  "id": "user uuid",
  "email": "user@example.com"
}
```

服务间共享同一个 `JWT_SECRET`。

### 15.2 文档访问

文档相关权限分为：

- owner：拥有完整读写和删除权限。
- share `read`：可读取和加入协同查看。
- share `write`：可读取、协同编辑、更新内容。

当前 `document-service` 的删除接口仅允许 owner 删除。

### 15.3 管理员

`users.role = 'admin'` 的用户可以访问管理员接口。管理员中间件位于 `services/user-service/src/middleware.ts`，部分管理员接口还会再次调用 `requireAdmin()` 查询数据库角色。

## 16. 核心业务链路

### 16.1 注册登录

```text
RegisterPage
  -> POST /api/user/verification-code
  -> user-service sends email
  -> POST /api/user/register
  -> user-service verifies code and creates user
  -> returns JWT
  -> AuthContext stores token and user
```

如果 SMTP 未配置，注册验证码发送不可用；Google 登录可作为替代登录方式，但需要配置 Google OAuth。

### 16.2 创建和编辑笔记

```text
Sidebar / NoteList
  -> NotesContext.createNote()
  -> POST /api/doc/notes
  -> MongoDB documents insert
  -> EditorPage opens /note/:id
  -> GET /api/doc/notes/:id
  -> WebSocket connects /ws/collab/:id
  -> TipTap + Yjs edits in realtime
  -> title/content autosave PUT /api/doc/notes/:id
```

需要注意：协同正文实时状态以 Yjs update 为准，HTTP 保存更多承担标题、兼容内容和离线同步职责。

### 16.3 PDF 到可编辑笔记

```text
EditorPage PDF input
  -> POST /api/doc/pdf/upload
  -> MinIO stores raw PDF
  -> ai-service /pdf/parse
  -> MinerU or PyMuPDF extracts markdown
  -> Chroma stores chunks
  -> MongoDB documents creates note
  -> MongoDB pdf_assets stores metadata
  -> frontend navigates to generated note
```

### 16.4 摘要和问答

```text
EditorPage
  -> POST /api/ai/summary?stream=true
  -> if text provided, summarize text
  -> if no text and noteId/documentId provided, search Chroma context
  -> stream SSE result

EditorPage
  -> POST /api/ai/chat?stream=true
  -> Chroma vector search by question
  -> LLM answers with retrieved context
  -> stream SSE result with sources
```

### 16.5 版本恢复

```text
VersionHistory
  -> POST /api/doc/notes/:id/versions/:versionId/restore
  -> document-service snapshots current state
  -> if selected version has yjsUpdate, write it to document_updates
  -> otherwise restore HTML content to documents
  -> frontend reloads page so Yjs provider reconnects to restored state
```

## 19. 常见问题和排查

### 19.1 Docker 前台启动后页面打不开

如果使用的是：

```bash
docker compose up --build
```

这是前台命令。终端停止、会话退出或命令结束后服务就不可用了。需要长期运行时使用：

```bash
docker compose up -d --build
docker compose ps
```

### 19.2 Google 登录尚未配置

原因通常是 `.env` 中：

```env
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=
```

为空。配置后还要确保 Google Console 中的 redirect URI 与当前模式一致：

- Docker/Nginx：`http://localhost/api/user/google/callback`
- Vite dev：`http://localhost:5173/api/user/google/callback`

### 19.3 邮件服务尚未配置

注册验证码依赖 SMTP。需要配置：

```env
SMTP_HOST=
SMTP_PORT=587
SMTP_SECURE=false
SMTP_USER=
SMTP_PASS=
MAIL_FROM=
```

### 19.4 MySQL 持久化 volume 缺列

如果登录或注册时看到类似 `Unknown column 'role' in 'field list'`，说明已有 MySQL volume 是旧 schema。当前用户服务启动时有 schema 修复逻辑，优先查看：

```bash
docker compose logs --tail=160 user-service
```

必要时检查 `services/user-service/src/db.ts` 的 `ensureUserSchema()`。

### 19.5 Nginx bind mount 状态异常

如果只有 nginx 失败，日志里出现 `/etc/nginx/conf.d/default.conf` 和 “not a directory”，可能是旧容器保留了错误 bind mount 状态。可重建 nginx 容器：

```bash
docker compose rm -f nginx
docker compose up -d nginx
```

### 19.6 `/ws/health` 不可用

协同服务没有 `/ws/health`。正确检查方式：

- HTTP：`collab-service:3004/health`
- WebSocket：`/ws/collab/:documentId`

### 19.7 Chroma 持久化丢失

Chroma 数据存储在 `/app/chroma_data`，通过 Docker volume `chroma-data` 持久化。如果 `docker compose down -v` 删除了 volume，索引数据会丢失，需要重新上传 PDF。

### 19.8 代理影响容器内本地访问

AI 服务访问外部 API 时使用 `httpx(..., trust_env=False)`，避免继承宿主机或容器环境中的代理变量。如果新增内部 HTTP 调用且出现莫名其妙的本地连接问题，优先检查代理变量和 `trust_env`。

## 20. 开发约定和扩展建议

### 20.1 新增 API

- 对外 API 优先挂在现有服务边界下，例如文档相关放入 document-service。
- 通过 Nginx 暴露时同步更新 `infra/nginx/default.conf`。
- 前端调用统一走 `apps/web/src/lib/api.ts`。
- 需要鉴权时使用 Bearer JWT。

### 20.2 新增 MongoDB 集合

- 在 `infra/database/mongodb/init.js` 补充集合和索引。
- 代码运行时仍应容忍集合不存在，MongoDB 会在首次写入时创建。
- 需要按用户查询的集合，应至少建立 `{ userId: 1, createdAt/updatedAt: -1 }` 类索引。

### 20.3 新增 MySQL 字段

- 更新 `infra/database/mysql/init.sql`。
- 同步更新 `services/user-service/src/db.ts` 中的 schema repair，避免已有 volume 运行时报错。

### 20.4 新增 AI provider

在 `services/ai-service/app/main.py` 中扩展：

- `provider_name()` 的可接受值。
- `ai_model_name()` 和 `ai_base_url()` 的默认模型、默认 base URL。
- `call_model()` 的非流式调用。
- `stream_model()` 的流式调用。

保持返回格式不变，前端就不需要改动。

### 20.5 新增离线变更类型

需要同时改：

- `apps/web/src/lib/offlineDb.ts` 的 `SyncChangeType` 和队列合并逻辑。
- `apps/web/src/contexts/NotesContext.tsx` 的 push 后处理。
- `services/sync-service/src/index.ts` 的 `SyncChange` 和 `/push` 处理。

## 21. 当前实现边界

- AI 默认 `mock` provider，适合离线演示，不代表真实大模型效果。
- PyMuPDF 回退模式主要做文本抽取，复杂版面、公式、表格效果有限。
- 分享权限已有基础读写模型，但复杂协作者邀请和团队空间仍可继续增强。
- 协同正文和 HTTP `documents.content` 并存，新增正文相关能力时要明确以哪一份为准。
- 离线同步以笔记 CRUD 为主，实时协同的离线合并不是完整 CRDT 离线方案。
- 当前缺少系统化自动化测试，重要改动建议至少跑构建、AI compileall 和容器内 smoke。

## 23. 面向开发的最小验收清单

完成一项较大改动后，建议至少验证：

- `npm run build` 通过。
- `python -m compileall services/ai-service/app` 通过。
- `docker compose up -d --build` 后 `docker compose ps` 核心服务为 `Up`。
- 网关能访问 `/api/user/health`、`/api/doc/health`、`/api/ai/health`、`/api/sync/health`。
- 注册/登录或 Google 登录入口未被破坏。
- 创建笔记、编辑保存、刷新恢复正常。
- 同一笔记两个窗口打开，协同输入能同步。
- PDF 上传能返回 `noteId` 并生成笔记。
- 摘要和问答能返回结果。
- 离线创建或编辑后恢复网络，能同步或进入明确冲突状态。
