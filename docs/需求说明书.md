# 软件系统需求说明书

## 1. 引言

### 1.1 编写目的

本文档旨在准确描述基于 AI 增强的协作式文档笔记系统的功能、非功能、设计约束、数据及交付等全部需求，统一项目开发、测试、部署、管理及终端用户的认知边界，作为系统开发、测试、验收与迭代的核心依据。

预期读者：项目开发团队、测试人员、部署运维人员、项目负责人、终端用户（学生/科研人员）。

### 1.2 背景（项目的来源和定义）

- 项目名称：基于 AI 增强的协作式文档笔记系统
- 用户单位：高校学生、科研人员等知识工作者
- 开发单位：项目开发团队

用户大致的要求和期望：解决 PDF 静态文件与动态笔记之间的隔阂，实现“输入（PDF）- 处理（AI）- 产出（Markdown 协作）”的闭环。支持多人实时协同编辑 Markdown 笔记，提供毫秒级同步、光标/选区显示及版本回溯。集成 AI 能力（摘要、润色、RAG 问答），帮助用户快速提取文献核心论点。支持本地优先的数据存储与离线编辑，保证数据安全与可用性。

### 1.3 定义

| 术语/缩写 | 解释 |
| --- | --- |
| Markdown | 一种轻量级标记语言，允许使用易读易写的纯文本格式编写文档，并可转换为 HTML。 |
| 协同编辑 | 多人同时在线编辑同一份文档，实时看到彼此的修改内容。 |
| CRDT | 无冲突复制数据类型（Conflict-free Replicated Data Type），用于实现分布式系统中的数据同步，避免冲突。 |
| RAG | 检索增强生成（Retrieval-Augmented Generation），从向量数据库中检索相关文档片段，再交由大模型生成回答。 |
| 流式输出 | AI 生成内容以逐字或逐句的方式实时推送给用户，减少等待感。 |
| Yjs | 高性能 CRDT 协同编辑框架。 |
| TipTap | 基于 ProseMirror 的富文本编辑器，原生支持 Yjs 协同。 |
| LlamaIndex | 专为 RAG 设计的框架，提供数据连接、索引、检索、问答流水线。 |
| Chroma | 轻量级向量数据库。 |
| MVP (Minimum Viable Product) | 最小可行性产品，指能满足核心用户需求并可用于早期交付验证的最简系统版本。 |

## 2. 任务概述

### 2.1 目标

本系统的总体目标是打造一个以 AI 为核心驱动的协作笔记平台，打破 PDF 静态文件与动态笔记之间的隔阂。具体目标包括：

1. PDF 价值转化：支持 PDF 上传与智能解析，自动提取标题、段落、表格、公式等结构，生成 Markdown 笔记草稿，减少用户手动重构的工作量。
2. 实时协同编辑：支持多人同时在线的毫秒级内容同步，提供光标/选区实时显示及版本历史回溯，解决协作冲突与延迟问题。
3. AI 辅助阅读：针对选中文本或整篇文档提供智能摘要、润色功能；基于个人笔记库或特定文档进行 RAG 问答，帮助用户快速获取核心论点。
4. 本地优先与跨设备：支持离线编辑与本地自动保存，数据不强制依赖云端；可选登录，允许用户在不注册的情况下使用本地功能。
5. 轻量部署：采用 Docker 微架构部署方案，能够在 2 核 4G 服务器资源下稳定运行。

### 2.2 用户特点

| 用户类型 | 特点 | 期望 |
| --- | --- | --- |
| 普通用户（学生/科研人员） | 日常阅读大量 PDF 文献，有做笔记和整理知识的需求；熟悉 Markdown 基础语法；希望减少重复劳动。 | 快速将 PDF 转为可编辑笔记；一键获得摘要；能够对自己的笔记库进行智能问答；支持离线使用。 |
| 协作者 | 常与同学或同事共同完成文献综述、项目报告；需要实时看到他人修改，避免版本混乱。 | 多人实时编辑同一文档；看到其他用户的光标和选区；能够回溯历史版本。 |
| 系统管理员 | 负责系统部署与日常运维，具备 Linux 和 Docker 基础知识。 | 能够管理用户、监控系统状态、配置 AI 服务的 API 密钥等参数。 |

### 2.3 假定和前提

- 用户具备基础的计算机操作能力，了解 Markdown 基本语法。
- 用户设备能够连接互联网（离线编辑时 AI 功能受限，但编辑功能正常）。
- 开发团队能够使用任务书中推荐的组件（Yjs + TipTap、MinerU/Marker、LlamaIndex + Chroma 等）。
- 系统部署环境满足最低资源要求（2 核 4G 服务器，已安装 Docker）。
- AI 服务依赖外部 API（如 DeepSeek），需要提供可用的 API 密钥，或本地部署轻量模型作为备选。

### 3.2 非功能性需求

#### 3.2.1 性能需求

- 协同同步延迟：在正常网络条件下（延迟 < 100ms），多用户编辑变更同步延迟不超过 200ms。
- 文档加载时间：打开一篇中等大小（5000 字）的 Markdown 文档，从点击到可编辑时间不超过 2 秒。
- PDF 解析响应：上传 10MB 以内的 PDF，解析并生成 Markdown 草稿的总耗时不超过 30 秒（典型学术论文）。
- AI 摘要/问答首字时间：流式输出下，用户发起请求后首字返回时间不超过 3 秒。
- 并发支持：系统支持至少 200 个同时在线用户，支持单个文档至少 8 人同时编辑。

#### 3.2.2 可靠性需求

- 数据持久性：用户笔记数据不因服务重启或崩溃而丢失；本地存储与云端存储互为备份。
- 崩溃恢复：编辑器或浏览器崩溃后，用户重新打开笔记，未保存的内容应能从本地自动恢复中找回。
- 协同一致性：在断开重连、网络抖动等场景下，最终所有协作者看到的内容必须一致（CRDT 保证无冲突）。

#### 3.2.3 可扩展性需求

- 支持新增 AI 模型、文件解析格式、协作功能。
- 支持用户量与数据量扩容。

#### 3.2.4 安全性需求

- 身份认证：使用 JWT 进行无状态认证，密码非明文存储。
- 数据隔离：用户只能访问自己创建或被明确共享的笔记；管理员无权查看用户原始笔记内容（仅能元数据管理）。
- 传输加密：所有客户端-服务端通信必须通过 HTTPS/WSS。
- 防注入：对用户输入的 Markdown 内容进行 XSS 过滤（前端+后端双重校验）。

## 5. 数据要求

### 5.1 逻辑描述

- 用户：包含用户 ID、用户名、密码哈希、角色、邮箱、注册时间。
- 笔记：包含笔记 ID、标题、Markdown 内容、创建者 ID、创建时间、更新时间。
- PDF 文档：包含 PDF ID、文件名、页数、上传者 ID、关联笔记 ID。
- 版本快照：包含快照 ID、笔记 ID、内容快照、创建者 ID、创建时间。
- 向量切片：包含切片 ID、笔记 ID、文本内容、向量嵌入、元数据。

### 5.2 输入输出要求

- PDF 上传：支持格式 .pdf，单个文件 <= 50MB（推荐 <= 20MB）。
- Markdown 导入/导出：支持导入 .md 文件，导出 .md 或 .html。
- AI 请求输入：用户自然语言或选中文本；输出为流式文本。

### 5.3 数据能力

- 支持单用户海量笔记存储与检索。
- 支持多人协同数据实时同步与一致性保障。
- 支持向量数据高效检索，支撑 AI RAG 功能。
- 支持数据本地导出、备份、离线存储与联网同步。

## 6. 其他要求

### 6.1 界面要求

- 整体风格简洁专业，适配学习/科研场景。
- Markdown 编辑实时渲染，多用户光标可视化展示。
- 功能入口清晰，操作步骤极简。

### 6.2 进度要求
