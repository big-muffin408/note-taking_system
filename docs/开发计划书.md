# 基于 AI 增强的协作式文档笔记系统开发计划书

## 1. 引言

### 1.1. 编写目的

明确“基于 AI 增强的协作式文档笔记系统”在整个开发生命周期中的工作内容、人员分工、时间进度、资源需求及风险管理策略。本文档旨在为 8 人开发团队提供清晰的行动依据，确保各项研发工作有序开展与高效推进，同时为项目利益相关者（如指导教师、团队成员等）提供项目执行的透明度与评估标准。

### 1.2. 背景

在知识爆炸的时代，学生和科研人员常常面临“收藏从未停止，阅读从未开始”的困境 。传统的静态 PDF 文献存在提取难、重构慢的“价值孤岛”问题，且多人协作时容易产生版本冲突和延迟 。 本项目旨在开发一套基于“边端协同”架构的 AI 协作笔记系统 。该系统采用“前端托管+云端数据+本地算力/模型服务商”的异构部署方案 ，打破 PDF 静态文件与动态笔记之间的隔阂，实现“输入(PDF)-处理(AI)-产出(Markdown 协作)”的闭环 。系统对标 Notion 级别的协作体验与 NotebookLM 级别的深度理解能力。

### 1.3. 定义

- **CRDT (Conflict-free Replicated Data Type):** 无冲突复制数据类型，本项目中基于 Yjs 实现多人实时协同编辑的底层技术保障。
- **RAG (Retrieval-Augmented Generation):** 检索增强生成，通过外挂知识库（如 PDF 解析后的向量切片）提升大模型回答准确性的技术。
- **MVP (Minimum Viable Product):** 最小可行性产品，指能满足核心用户需求并可用于早期交付验证的最简系统版本。

## 2. 项目概述

### 2.1. 工作内容

本项目的核心任务是开发一套支持多人实时协同的 Markdown 笔记系统，并深度集成 AI 能力。主要工作包括：

1. 搭建基于 Yjs 和 TipTap 的实时渲染块级 Markdown 编辑器。
2. 集成 MinerU 实现 PDF 的结构化解析与 Markdown 转换。
3. 利用 LlamaIndex 和 Chroma 构建 RAG 问答流，实现基于个人笔记库的 AI 问答与全文摘要。
4. 完成基于 Docker 的微服务架构部署，涵盖用户、协同、文档、AI 等独立服务。

### 2.2. 项目主要成员

本项目由 8 人开发团队承接，成员具备全栈开发、AI 提示词工程、UI 设计及 DevOps 运维能力，涵盖从逻辑分析、技术架构到前端交互与后端并发处理的综合技术栈。

### 2.3. 产品及成果

#### 2.3.1. 程序

- **前端 Web 应用:** 基于 React/Vue 开发，支持协作编辑与 AI 交互交互的浏览器端程序。
- **后端微服务集群:** 包含用户服务、文档服务、协同服务及 AI 服务等后端程序镜像。
- **数据库初始化脚本:** MySQL、MongoDB 及 Qdrant 的建表与索引脚本。

#### 2.3.2. 文档

- 《软件需求规格说明书》：定义系统功能细节及 AI 触发逻辑。
- 《系统架构与 API 设计文档》：包含微服务架构图、数据库说明及接口规范。
- 《测试分析报告》：包含自动化测试结果及多人协作一致性校验结果。
- 《项目任务书》：明确项目目标、任务范围、交付成果、责任分工与验收标准。
- 《项目开发计划书》：规划项目整体进度安排、资源配置、技术路线等，指导项目实施。

#### 2.3.3. 服务

- **部署上线:** 在 2核4G 服务器上完成微服务环境搭建与内网穿透配置。
- **运行支持:** 在学期项目周期内提供系统监控、日志分析及 bug 修复服务。

#### 2.3.4. 非移交产品

- 内部开发环境配置脚本与测试桩代码。
- UI/UX 高保真设计源文件（Figma 工程）。

### 2.4. 验收标准

- **功能验收:** 满足 MVP 核心功能，即支持 CommonMark 语法的实时协作、PDF 拖拽提取文本分段、以及一键生成摘要。
- **性能验收:** 多人同文档编辑达到毫秒级同步，光标显示无明显卡顿。
- **稳定性验收:** 在服务器的资源限制下，各个 Docker 容器能平稳运行，无内存溢出导致的服务崩溃。

### 2.5. 完成项目的最迟期限

**学期末** （具体日期依据校历与课程结课要求确定）

## 5. 专题计划要点

### 5.1. 沟通计划

- **周会同步:**
  - **时间:** 每周固定时间（如周日晚）。
  - **内容:** 汇报本周进度、展示阶段成果、讨论阻塞问题 (Blockers) 及分配下周任务。
  - **方式:** 线下会议或语音会议。
  - **对象:** 全体团队成员。
  - **负责人:** 需求分析负责人兼项目经理。
- **代码与技术评审:**
  - **时间:** 核心模块合并入主分支前。
  - **内容:** 检查代码规范、架构合理性及是否存在性能隐患。
  - **方式:** Git Pull Request 异步评审与线上讨论。
  - **对象:** 涉及对应模块的开发人员与代码负责人。
  - **负责人:** 代码负责人。

### 5.2. 质量保证计划

- **质量目标：** 确保系统在 MVP 验收时，多人协同的低延迟、无丢失，且大模型问答具备较高的文档关联准确性。
- **质量活动安排：**
  - **接口与单元测试:**
    - **时间:** 编码阶段并行。
    - **内容:** 对用户权限、文档 CRUD 等后端 API 进行测试。
    - **检查方式/依据:** 自动化测试脚本覆盖率。
    - **负责人/参与人:** 代码开发组员、测试负责人。
  - **并发一致性演练:**
    - **时间:** 第二次迭代完成前后。
    - **内容:** 模拟 3-5 人同时对同一文档的同手段落进行复杂修改与撤销。
    - **检查方式/依据:** 观察最终文本的一致性，检查 VersionHistory 记录。
    - **负责人/参与人:** 测试负责人主导，开发成员配合演练。
  - **资源占用压测:**
    - **时间:** 部署上线阶段。
    - **内容:** 监控微服务全开时的 2C4G 服务器内存状态。
    - **检查方式/依据:** Linux 系统监控面板 (如 htop) 或 Docker Stats，确保不发生 OOM (Out of Memory)。
    - **负责人/参与人:** 部署负责人。
